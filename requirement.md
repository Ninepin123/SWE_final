Chapter 1 Project Scope
1.1 Identification
System Name:
National University of Kaohsiung Scholarship Application and
Management System (NUKSAMS)
Subsystem Name:
Account and Authority Management Subsystem(AAS)
Scholarship Management Subsystem(SMS)
Student Application Subsystem(SAS)
Teacher Recommendation Subsystem(TRS)
Review and Award Subsystem(RAS)
Notification and Communication Subsystem(NCS)
Database Subsystem(DBS)
1.2 Overview
本系統主要目標是建立一套整合式的獎助學金申請與管理平台，讓學生、獎助學
金提供單位、推薦教師、審核人員與系統管理員能夠在線上完成申請、推薦、審查、
通知與管理作業。
NUKSAMS 將傳統紙本或 Email 往返的申請流程改為線上化與自動化。學生可依
科系、GPA、年級或其他條件查詢可申請的獎助學金，並在線上填寫申請資料、上傳文
件與追蹤進度。獎助學金提供單位可建立獎助學金項目、設定條件、名額、金額與截
止日期；教師與審核人員則可依權限查看資料，進行推薦與審查。
1.3 System Description
本系統 NUKSAMS 的主要功能是讓學生、老師、獎助單位、審查單位與系統管理
員能透過網路完成獎助學金申請、推薦、審查、通知與管理作業。基本上，本系統功
能可分為七個方面：
1. 帳號與權限管理
提供使用者登入、角色權限控管、帳號 CRUD、單位資料隔離與操作紀錄。
2. 獎助學金資料管理
提供獎學金基本資料、申請條件、名額、截止日期、分類與標籤管理。
3. 學生申請管理
1

提供學生查詢可申請獎學金、線上填寫申請表、上傳文件、修改個人資料、詢
申請進度與補件。
4. 教師推薦管理
提供老師查看被邀請推薦的學生資料、撰寫推薦信、草稿暫存、提交推薦信與
查看負責案件進度。
5. 審查與核發管理
提供審查單位查看申請資料、自動排序、審查通過或不通過、發送補件要求、
核定獎學金結果與產生統計報表。
6. 通知與溝通管理
提供 Email、站內通知、公告、留言互動、補件提醒、截止提醒與系統異常通
知。
7. 資料庫管理
管理整個系統中使用者、獎學金、申請案、文件、推薦信、審查紀錄、通知與
系統紀錄的存取。
1.4 Subsystem AAS Description
帳號與權限管理子系統(AAS)是系統管理員與所有使用者進入系統的基礎子系統，
主要用來管理登入驗證、帳號資料、角色權限與資料存取範圍。其主要功能如下：
1. 使用者登入管理 Login Management
提供學生、老師、獎助單位、審查人員與系統管理員登入系統的功能，並驗證
其身份與角色。
2. 帳號管理 Account Management
提供系統管理員新增、修改、刪除與查詢使用者帳號，例如學生帳號、老師帳
號、獎助單位帳號與審查人員帳號。
3. 角色與權限管理 Role and Permission Management
根據使用者身份限制其可使用的功能。例如學生只能查看自己的申請狀態，老
師只能查看自己負責推薦的案件，審查單位只能查看所屬單位的申請資料。
4. 單位資料隔離 Unit Data Isolation
限制不同獎助單位之間不得互相查看申請資料，避免資料外洩。系統管理員可
依權限進行全域管理。
5. 管理操作紀錄 Audit Log Management
紀錄帳號新增、刪除、修改、審查操作與權限變更等重要行為，方便日後追蹤
與稽核。
6. 系統維運監控 System Monitoring
提供管理員查看線上人數、伺服器負載、系統異常狀態等資訊，並在系統過載
或異常時觸發警示。

1.5 Subsystem SMS Description
獎助學金資料管理子系統(SMS)是獎助單位與系統管理員用來建立與維護獎助學
金資料的子系統。此子系統主要負責申請開始前的設定工作。其主要功能如下：
1. 獎助學金基本資料管理 Scholarship Information Management
提供獎助單位新增、修改、刪除獎學金資料，例如獎學金名稱、年度、金額、
說明、聯絡人、Email、電話、地址與網站。
2. 申請條件管理 Eligibility Criteria Management
提供獎助單位設定申請門檻，例如 GPA、科系限制、家庭狀況、年級、身份條
件或其他資格條件。
3. 申請期限與名額管理 Deadline and Quota Management
提供獎助單位設定申請開始時間、截止時間與申請名額。當截止時間到達或名
額額滿時，系統自動關閉申請通道。
4. 獎助學金分類與標籤管理 Category and Tag Management
提供系統管理員或獎助單位依照來源或性質將獎學金分類，例如校內、政府、
私人贊助、清寒、成績優良等。
5. 可申請項目篩選資料提供 Scholarship Matching Support
此子系統提供獎學金條件資料給學生申請子系統，使學生首頁能顯示符合資格
或尚未截止的可申請項目。
1.6 Subsystem SAS Description
學生申請子系統(SAS)是學生用來查詢獎助學金、填寫申請資料、上傳文件與追
蹤申請進度的子系統。其主要功能如下：
1. 個人資料管理 Profile Management
提供學生註冊、查看、修改與更新個人基本資料。但學號、身份類別等核心
資料不可由學生任意修改。
2. 獎助學金查詢與篩選 Scholarship Search and Filtering
提供學生依照科系、GPA、申請資格、獎學金類別、截止日期等條件查詢獎助
學金，並在首頁顯示目前可申請的項目。
3. 線上申請 Online Application
提供學生線上填寫申請表單，完成無紙化申請流程。
4. 文件上傳 Document Upload
提供學生上傳成績單、自傳、證明文件或其他申請附件，並限制檔案格式與
大小。
5. 申請進度查詢 Application Status Tracking
提供學生查看所有申請案件的狀態，例如審核中、需補件、已通過、未通過。
6. 補件管理 Supplement Submission
當審查單位要求補件時，學生可於期限內上傳缺少的文件，並由系統通知審
查人員。
3

7. 申請截止後鎖定 Application Locking
申請截止後，學生不得再修改申請資料或重新上傳文件，以維持申請公平性。
1.7 Subsystem TRS Description
教師推薦子系統(TRS)是老師用來查看學生資料、撰寫推薦信與追蹤推薦案件進
度的子系統。其主要功能如下：
1. 推薦案件列表 Recommendation Request List
提供老師查看目前需要撰寫推薦信的學生名單，並可依姓名、申請時間、截止
日期或申請狀態排序與篩選。
2. 學生完整資料檢視 Student Profile Review
老師點擊學生姓名後，可查看學生基本資料、成績單、修課紀錄、自傳與相關
經驗，以便撰寫推薦信。
3. 推薦信撰寫與編輯 Recommendation Writing and Editing
提供老師線上撰寫、修改與提交推薦信。
4. 推薦信草稿暫存 Draft Saving
老師可在尚未完成推薦信時暫存草稿，之後再繼續編輯。
5. 推薦信權限控管 Recommendation Privacy Control
推薦信內容僅限撰寫老師與審查人員查看，學生只能看到推薦信是否已提交，
不能查看推薦信內容。
6. 推薦案件進度查詢 Recommendation Progress Checking
老師可查看自己負責案件目前的處理狀態，但不得查看非自己負責的案件。
1.8 Subsystem RAS Description
審查與核發管理子系統(RAS)是審查單位與獎助單位用來處理申請案件、審核資
料、核定結果與產生統計資料的子系統。其主要功能如下：
1. 申請案件查詢 Application Query
提供審查人員依年度、學生姓名、科系、申請狀態、獎學金類別等條件查詢申
請資料。
2. 申請資料檢視 Application Review
提供審查人員查看學生基本資料、成績證明、上傳文件、推薦信與申請內容。
3. 自動排序 Ranking Management
系統可依 GPA、申請條件或審查標準自動排序申請者，協助審查人員快速處理
案件。
4. 審查結果管理 Review Decision Management
提供「通過」與「不通過」操作，並記錄審查人員、審查時間與審查結果。
5. 補件通知管理 Supplement Request Management
若申請資料缺漏，審查人員可透過系統發送補件通知，並設定補件期限。學生
補件後，系統通知審查人員。
4

6. 核發名單管理 Award List Management
審查完成後，系統產生獎助學金核發名單，包含得獎學生、核定金額與申請年
度。
7. 年度統計與報表 Reporting and Statistics
系統可統計年度發放總金額、得獎人數、申請人數與通過比例，並支援匯出
PDF 或 Excel 報表。
8. 單位資料存取限制 Unit-Based Access Control
審查單位只能查看自己單位相關的申請資料，不能查看其他單位的案件。
1.9 Subsystem NCS Description
通知與溝通子系統(NCS)是所有子系統共用的支援子系統，用來處理站內通知、
Email 通知、公告、留言與問題回報。其主要功能如下：
1. Email 與站內通知 Email and In-System Notification
當獎學金即將截止、推薦信尚未完成、申請狀態異動、補件期限將至或審查結
果產生時，系統自動發送通知。
2. 截止提醒 Deadline Reminder
系統可在截止前通知學生、老師或審查單位，避免錯過申請、推薦或補件期限。
3. 公告管理 Announcement Management
系統管理員可發布全域公告，使所有使用者即時得知重要訊息。
4. 留言互動 Message Board
提供學生與獎助單位或審查單位在系統內直接留言溝通，減少 Email 往返。
5. 問題回報 Issue Reporting
提供使用者回報系統問題或提出意見，管理員可查看、回覆與追蹤處理進度。
6. 系統異常通知 System Alert
當系統過載、伺服器異常或監測指標異常時，通知系統管理員。
1.10 Subsystem DBS Description
資料庫子系統(DBS)用來管理整個系統的資料存取，包含新增、修改、刪除、查
詢與資料一致性維護。所有子系統之間的資料傳遞皆透過資料庫子系統或其資料存取
服務處理。其主要功能如下：
1. 使用者資料管理 User Data Management
儲存學生、老師、審查人員、獎助單位與系統管理員的帳號與角色資料。
2. 獎助學金資料管理 Scholarship Data Management
儲存獎學金名稱、年度、金額、條件、名額、截止日期、分類與標籤。
3. 申請資料管理 Application Data Management
儲存學生申請表、申請狀態、上傳文件、補件紀錄與申請時間。
4. 推薦信資料管理 Recommendation Data Management
儲存老師撰寫的推薦信、草稿、提交狀態與權限設定。

高雄大學需求的獎助學金申請與管理系統。

7.1.3 Operational Concepts and Scenarios

7.1.3.1 Recommendation Request List Module Use Cases
No.: UC028
Name: 查看推薦案件列表（View recommendation request list）
Actors: 教師
Goals: 查看目前需要處理的推薦案件
Preconditions:教師必須已登入，且已有學生邀請或系統指派推薦案件。
Summary:教師查看待撰寫推薦信的學生名單，並可依姓名、申請時間、截止日期或狀
態篩選。
Step:
Actor System
1. 教師選擇「推薦案件列表」。 2. 系統查詢教師負責的推薦案件。
3. 教師查看案件清單。 4. 系統顯示學生姓名、申請項目、截止
日期與狀態。
5. 教師使用排序或篩選條件。 6. 系統更新顯示結果。
7.1.3.2 Student Profile Review Module Use Cases
No.: UC029
Name: 檢視學生完整資料（Review student profile）
Actors: 教師
Goals: 查看學生資料以撰寫推薦信
Preconditions:教師必須已登入，且該學生屬於教師負責推薦案件。
Summary:教師查看學生基本資料、成績單、修課紀錄、自傳與相關經驗。
Step:
Actor System
1. 教師於推薦案件列表選擇學生。 2. 系統檢查教師是否具有查看權限。
3. 教師查看學生資料。 4. 系統顯示基本資料、成績單、自傳與
相關文件。
5. 教師完成資料檢視。 6. 系統維持資料權限控管。

7.1.3.5 Recommendation Privacy Control Module Use Cases
No.: UC034
Name: 控管推薦信查看權限（Control recommendation privacy）
Actors: 系統、教師、審查人員、學生
Goals: 確保推薦信內容僅授權者可查看
Preconditions:推薦信資料已存在，且使用者角色已設定。

3. 教師或審查人員要求查看內容。 4. 系統允許顯示推薦信內容。
5. 學生要求查看推薦信。 6. 系統僅顯示推薦信是否已提交，不顯
示內容。

2. 使用方式和過程
在這一次軟體需求分析的報告當中的討論過程中，我們小組先對系統的提出自己
的想法，並且整理出一些基本的功能，以及相關使用情境和可能的使用者，並一起討
論我們所遇到的困難和需要解決的問題。一開始是用我們自己的理解，先初步把系統
功能的架構和操作流程大致寫出來，把它當作我們的初稿。
在我們完成初稿之後，我們再利用 AI 相關工具幫忙檢查我們所製作的內容，然
後看有沒有不夠清楚或者是不完整的地方，像是功能描述是不是太模糊、整個流程合
不合理，或是有沒有漏掉一些重要的需求之類的問題，我們透過 AI所提供的建議，
並且自己查看過一遍，再的內容修改一下。
我們不只用一種AI，我們利用不同的 AI 來進行交叉比對，因為不同AI工具給
的建議有時候會不太一樣，所以我們會去比較哪一些建議比較符合我們的專題，再跟
組員討論要不要採用到我們的專題，而不是直接全部生成的內容都採納，因為AI生成
的東西有些不太適合我們的內容。
在整個製作和修改報告的過程中，有些地方如果 AI 的建議和我們原本的設計方
向有不太一樣的地方，我們也會回頭來去做檢查檢查，再做細部的調整，讓整份報告
可以依照我們的方向來製作，而不是全部透過AI，AI只是我們製作報告的輔助工具。
